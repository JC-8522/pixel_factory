# Runtime Adapter Design

## Purpose

Runtime adapters isolate the rest of the app from how agents are controlled. The app should be able to use mock agents, spawned Codex CLI agents, attached Codex sessions, MCP orchestration, or future providers through one interface.

## Interface

```ts
export interface AgentRuntime {
  readonly kind: AgentRuntimeKind;

  discoverAgents(): Promise<DiscoveredAgent[]>;

  spawnAgent(input: SpawnAgentInput): Promise<AgentSession>;

  sendMessage(sessionId: string, message: string): Promise<void>;

  stopAgent(sessionId: string): Promise<void>;

  restartAgent?(sessionId: string): Promise<AgentSession>;

  streamEvents(sessionId: string): AsyncIterable<AgentEvent>;
}
```

## Runtime Kinds

```ts
type AgentRuntimeKind =
  | "mock"
  | "codex_cli_spawned"
  | "codex_cli_attached"
  | "mcp"
  | "external";
```

## Spawn Input

```ts
type SpawnAgentInput = {
  agentId: string;
  profileId?: string;
  profileSnapshot?: AgentProfileSnapshot;
  name: string;
  role: string;
  workingDirectory: string;
  initialTask: string;
  modelProfile?: string;
  skillIds: string[];
  skillPromptContext: string;
  autoRunMode: "manual" | "auto";
  permissionMode: "ask" | "workspace" | "readonly";
};
```

`role` may be `Manager Agent`. This means the spawned agent receives planning and coordination prompt context, but it does not receive human manager authority. Permission decisions and app settings remain controlled by the human user.

`profileSnapshot` is generated by the main process from the selected Agent Profile at creation time. Runtime adapters should use the snapshot to build prompt context, tool/capability boundaries, startup workflow instructions, validation expectations, communication style, risk tolerance, output preferences, and visual identity metadata where relevant.

Agent Packs are not runtime adapters. They are package sources that can install profiles, skills, assets, and validation metadata. Installed profiles from Agent Packs still create ordinary agent instances through the same `AgentRuntime` interface.

## Event Flow

1. Runtime adapter emits raw runtime events.
2. Main process normalizes events into `AgentEvent`.
3. Status machine maps events into agent statuses.
4. Main process persists messages, sessions, and timeline events.
5. Main process broadcasts sanitized updates to renderer.
6. Renderer stores update UI and PixiJS sprites.

## Mock Runtime

Mock runtime is required for MVP development.

Responsibilities:

- create deterministic sessions,
- emit predictable status changes,
- stream fake message chunks,
- simulate command events,
- simulate errors,
- support stop behavior,
- enable unit and UI tests without Codex installed.

## Codex CLI Spawned Runtime

Spawned runtime controls app-created local Codex sessions.

Responsibilities:

- build safe process arguments,
- start Codex from main process,
- set working directory,
- pass initial task and profile settings,
- include assigned skill context in prompts,
- capture stdout and stderr,
- parse runtime signals into events,
- support stop and restart,
- report process exit status.

All command execution must pass through security hooks before it is allowed to run.

## Attached Runtime

Attached runtime is V2 unless a reliable local signal is available earlier.

Potential sources:

- running Codex CLI processes,
- local session files,
- local logs,
- MCP session metadata.

Attach mode may be read-only at first. It must not pretend to control sessions that cannot safely accept messages or stop requests.

## MCP Runtime Bridge

MCP support should be an extension point, not a hard dependency for MVP.

Responsibilities:

- define typed bridge contracts,
- map MCP tool/session events into `AgentEvent`,
- preserve the same renderer-facing behavior,
- fail gracefully when MCP integration is not configured.

## Status Mapping

Runtime adapters may emit detailed events, but visible status is owned by the status machine.

| Visible Status | Typical Source Event |
| --- | --- |
| `idle` | no active session or waiting without task |
| `thinking` | prompt accepted, response pending |
| `running_command` | command started |
| `reading_files` | file read signal or log pattern |
| `editing_files` | file write/edit signal or log pattern |
| `waiting_user_input` | runtime requests approval or user input |
| `error` | stderr/error event or failed session |
| `completed` | task/session completed cleanly |
| `stopped` | user stopped the session |

## Safety Integration

Runtime adapters must not bypass safety.

- Commands requested by app-controlled agents go through `safeCommandGate`.
- Risky commands pause execution until permission is decided.
- Denied commands emit `command_denied` timeline events.
- Logs must pass through secrets redaction before renderer broadcast.

## Failure Rules

- If spawn fails, create an `error_occurred` event and set agent status to `error`.
- If process exits by user stop, set status to `stopped`.
- If process exits normally, set status to `completed`.
- If event streaming fails, keep persisted session data and show a recoverable error.
- If attach mode cannot control a session, expose read-only capabilities only.

## Testing Requirements

- Mock runtime tests must cover all event types.
- Codex CLI runtime tests should use a fake child process adapter.
- Status machine tests must cover every visible status.
- IPC runtime tests must verify invalid IDs and payloads are rejected.
- Renderer tests should use mock preload APIs, not real child processes.
